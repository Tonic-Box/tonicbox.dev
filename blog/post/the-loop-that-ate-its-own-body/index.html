The loop that ate its own body

2026-07-15 · TonicBox · YABR

I work on YABR, a bidirectional Java (de)compiler and transformation framework built around a stack of representations - bytecode, SSA, a structured IR, source - that a program can move freely across. Two of those layers matter here. The SSA IR is close to the bytecode: a control-flow graph of basic blocks, every value assigned once. The structured IR is close to source: a tree of expressions and statements. The bug lived in the pass between them, the one that lifts SSA into the structured IR - reconstructing if/else, while, for, the vocabulary a human would have written before javac flattened it into jumps.

Lately I've been building a transpiler on top of it: feed it a .jar, get a program in another language back out. There's a proper test suite behind all this, but to stress test it and help with hammering down JNI in transpilation, I wrote a simple game using jMonkey in Java. If any of it breaks, some method came out of the reconstructor subtly wrong, and I get to go find it.

Nothing, specifically

The game booted. The scene drew. I could walk around with WASD, the mouse looked where I pointed it, boxes fell and stacked and behaved like boxes. By every measure that throws an exception, the port was fine.

Except the debug overlay was gone. No FPS counter, no triangle count, none of the little white text jMonkeyEngine paints in the corner when you enable stats. Just... nothing there.

Missing text is a miserable thing to debug, because there are a hundred boring reasons for it and one interesting one. The boring ones came first:

  • Was the font atlas loading? I dumped the texture. It was there, glyphs and all.
  • Was the material getting a null texture? No - the same font renders fine if I hand it a string manually.
  • Was the overlay z-ordered behind something, or scaled to zero, or off-screen? I moved it, recolored it, made it enormous. Still nothing.

I spent that evening convinced it was the font. It wasn't.

Zero of something

The break came when I stopped looking at why the text wasn't visible and started looking at what the text actually was. jME builds on-screen text as geometry - each glyph is two triangles, positioned from the font's layout tables. So I reached into the BitmapText after it was supposed to have laid itself out and asked it how many vertices it had.

Zero.

Not garbage, not off-screen. Zero. The overlay was rendering perfectly; it just had nothing to render. So this was never a graphics bug, it was a data bug. Something upstream built an empty mesh.

The glyph geometry gets assembled in a jME internal called BitmapTextPage.assemble(). The original, roughly:

void assemble(Letters quads) {
    // ... reset state, size the buffers ...
    quads.rewind();

    while (quads.nextCharacter()) {
        if (quads.isPrintable() && quads.getCurrentPageIndex() == page) {
            pageQuads.add(quads.getQuad());
        }
    }

    // build the vertex/index buffers from pageQuads
    // ... write positions, tex coords, indices ...
}

pageQuads comes out empty only if nothing was printable, which, for the string "60 fps", is absurd.

So I pulled the version the reconstructor had produced for that same method. And there it was.

The reconstruction

Here's what came out the other end, cleaned up for readability:

void assemble(Letters quads) {
    // ... reset state, size the buffers ...
    quads.rewind();

    do {
        if (quads.nextCharacter()) {
            continue;
        }
        // build the vertex/index buffers from pageQuads
        // ... write positions, tex coords, indices ...
        return;
    } while (!quads.isPrintable());

    pageQuads.add(quads.getQuad());
}

It's a genuinely impressive piece of wrongness. It compiles. It runs. The control flow is valid. It's just been turned inside out:

  • The while became a do/while.
  • The mesh-building code - the part that was supposed to run after the loop - got pulled inside it, onto the very first exit path.
  • The real body of the loop, the pageQuads.add(...), got shoved out past the end of the loop, where it's dead code that never runs.

Trace it: first iteration, nextCharacter() returns true, continue. Second call returns false eventually, we fall into the mesh-building block, and return. We build the mesh from a pageQuads that nobody ever added to. Empty mesh. Zero vertices. No FPS counter.

The bug wasn't in rendering, or fonts, or buffers. The reconstructor had looked at a while loop and confidently rebuilt it as a do/while, and in doing so swapped the loop's body with its continuation. Everything downstream was working exactly as designed on top of a lie.

Now I had to find out why.

How a loop gets its shape back

Bytecode doesn't have loops. It has conditional jumps. while (cond) body and do body while (cond) can compile to almost the same instructions; the difference is where the test sits relative to the back-edge. Reconstructing the shape means looking at the CFG and deciding which structured form the jumps came from.

The pipeline for that is textbook:

  • Build the dominator tree.
  • Find back-edges - an edge A -> B where B dominates A. Every back-edge is a loop closing on itself.
  • For each back-edge, compute the natural loop: the header B, the tail A, and every block that can reach A without going through B.
  • Classify the loop - top-tested (while), bottom-tested (do/while), counted (for) - from where the condition lives.

Step 4 is where mine went wrong, but the root cause was hiding in step 3.

The distinguishing move for a do-while is finding what I call a conditional latch: a block inside the loop whose branch is the loop's bottom test - one arm jumps back to the header, the other arm leaves the loop. If a loop has exactly one such block, it's bottom-tested. The check is small and, I thought, obviously correct:

private IRBlock findConditionalLatch(IRBlock header, Loop loop) {
    IRBlock latch = null;
    for (IRBlock pred : header.getPredecessors()) {
        if (!loop.contains(pred)) {
            continue;               // predecessor isn't part of this loop
        }
        if (latch != null) {
            return null;            // more than one in-loop predecessor -> not a clean bottom test
        }
        latch = pred;
    }
    // ... latch must end in a branch, one arm to header, one arm out of the loop ...
}

Everything hangs on that if (latch != null) return null;. Two or more in-loop predecessors feeding the header means this can't be one clean bottom test, so bail out and treat it as a plain top-tested loop. Hold onto that guard.

Two back-edges walk into a header

assemble's loop body has this shape:

while (quads.nextCharacter()) {
    if (quads.isPrintable() && quads.getCurrentPageIndex() == page) {
        pageQuads.add(quads.getQuad());
    }
}

That body has two skip paths: one for a non-printable character, one for a character on the wrong page. When either fires, control jumps back to the header to test the next character. Two back-edges to the same header.

And here's the thing I'd gotten wrong two steps earlier, in the natural-loop code:

// old identifyLoops(): one Loop per back-edge
for (Map.Entry<IRBlock, Set<IRBlock>> entry : backEdges.entrySet()) {
    IRBlock tail = entry.getKey();
    for (IRBlock header : entry.getValue()) {
        loops.add(new Loop(header, findNaturalLoop(header, tail)));
    }
}

One Loop object per back-edge. A loop with two continue paths produced two Loops, both with the same header, each containing a different subset of the body - because the natural loop of back-edge #1 doesn't necessarily include the blocks that only back-edge #2 reaches.

Downstream, when the classifier wanted "the loop for this header," it grabbed the first one it found:

private Loop findLoopWithHeader(IRBlock block) {
    for (Loop loop : loopAnalysis.getLoops()) {
        if (loop.getHeader() == block) {
            return loop;      // whichever one happens to be first
        }
    }
    return null;
}

So findConditionalLatch ran against a loop missing half its own body. The header really had two in-loop predecessors, but one of them lived in the blocks this undersized Loop didn't know about. The latch check saw exactly one, latch != null never tripped, and the guard that should have said "too many back-edges, not a bottom test" stayed silent.

One surviving predecessor, ending in a branch, one arm looping, one arm leaving. Textbook conditional latch. The classifier declared do/while with total confidence, handed the reconstructor a header and a latch that described a loop half the real size, and the reconstructor faithfully did what it was told: put the code after the "latch" outside the loop, put the header's own conditional inside as a guard, and turn the whole thing inside out.

Every stage did its job correctly. They were just all reasoning about a loop that didn't exist.

The fix is boring, which is how you know it's right

A header with several back-edges has one loop, not several. The natural loops of the individual back-edges are subsets of the real thing; you want their union. So: group back-edges by header, union their natural loops, emit one Loop per header.

// group tails by the header they jump to
Map<IRBlock, Set<IRBlock>> headerToTails = new LinkedHashMap<>();
for (var entry : backEdges.entrySet()) {
    IRBlock tail = entry.getKey();
    for (IRBlock header : entry.getValue()) {
        headerToTails.computeIfAbsent(header, k -> new LinkedHashSet<>()).add(tail);
    }
}

// one loop per header: union of every back-edge's natural loop
for (var entry : headerToTails.entrySet()) {
    IRBlock header = entry.getKey();
    Set<IRBlock> blocks = new HashSet<>();
    for (IRBlock tail : entry.getValue()) {
        blocks.addAll(findNaturalLoop(header, tail));
    }
    loops.add(new Loop(header, blocks));
}

The one wrinkle: nesting still has to work. A block inside two genuinely nested loops needs to resolve to the innermost one. So after building the merged loops, map each block to the smallest loop that contains it, and getLoop(block) still returns the tightest enclosing loop.

With the union in place, assemble's header now sees both of its in-loop predecessors. findConditionalLatch finds two, hits latch != null, returns null, and the loop is classified as the plain top-tested while it always was. The mesh-building code goes back after the loop. pageQuads gets its quads. The FPS counter came back.

What I took away from it

The reason this one stuck with me is that at no point did anything fail. There was no exception, no assertion, no malformed graph. A block-partition bug in step three quietly starved a heuristic in step four, and the heuristic gave a confident, well-formed, completely wrong answer. Every layer above it built correctly on top of that answer. The only visible consequence, six layers up, was that some white text had zero vertices.

If there's a lesson, it's that a heuristic is only as good as the facts you feed it. findConditionalLatch was being fed a lie by a data structure two stages upstream. The check itself was fine. The loop it was checking wasn't a loop.

I now have a regression test that's just a while loop with two continues followed by code that has to run after it. It's a stupidly small method. It would have caught an entire evening of blaming the font.
